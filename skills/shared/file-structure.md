# File Structure Convention

All harness project state lives under `docs/project/`. This is a convention — the harness detects its presence to activate project-aware behavior.

## Structure

```
docs/project/
├── vision.md                          # Project identity and boundaries
├── roadmap.md                         # Module → work item index (the global map)
├── architecture.md                    # Domain boundaries, tech conventions (if needed)
├── quality.md                         # Quality scores, tech debt (generated by /project-health)
├── external-deps.md                   # External system dependencies (if any)
│
├── specs/                             # Current system behavior (source of truth)
│   ├── index.md                       # Spec index
│   └── {domain}/{spec-name}.md        # One spec per completed feature
│
├── exec-plans/                        # Execution plans per module
│   ├── active/{module-id}.md          # Currently in progress
│   ├── completed/{module-id}.md       # Finished and archived
│   └── abandoned/{module-id}.md       # Dropped (with reason)
│
├── decisions/                         # Key decision records
│   └── {date}-{topic}.md
│
├── assets/                            # Non-code deliverable tracking
│   ├── manifest.md                    # Master asset list with status
│   └── {domain}/                      # Per-domain asset specs (optional)
│
├── ops/                               # Operational/recurring activities (if in ops mode)
│   └── {activity}.md
│
├── .current-work.md                   # Current work item context (transient)
└── .system-map.md                     # Brownfield: existing system audit (if applicable)
```

## File Presence as Feature Flag

The harness uses file presence to activate behavior:

| File | Activates |
|------|-----------|
| `docs/project/` directory exists | All harness project-aware behavior |
| `docs/project/.current-work.md` exists | Execution layer reads it as context; session-start hook injects it |
| `docs/project/architecture.md` exists | Architecture constraints injected into brainstorming and commit checks |
| `docs/project/.system-map.md` exists | Brownfield-aware behavior in relevant skills |
| `docs/project/assets/manifest.md` exists | Asset tracking in project-next, project-done, project-status |
| `docs/project/external-deps.md` exists | External dependency checks in project-health |
| `docs/project/ops/` directory exists | Ops mode available in project-status |

**If `docs/project/` does not exist**, the harness behaves identically to original Superpowers. This is the backward compatibility guarantee.

## Roadmap Format

```markdown
# Roadmap

## Milestones
- **v0.1 Name**: M1 + M2 → what's deliverable at this point
- **v0.2 Name**: + M3 → description

---

## M1: Module Name [status: planned] [milestone: v0.1]

Brief description.

- [ ] F1.1: Feature name [feat, P0, deps: —]
- [ ] F1.2: Feature name [feat, P0, deps: F1.1]
- [ ] I1.1: Infra item [infra, P0, deps: —]
- [decision: summary] S1.1: Spike name [spike, deps: —]
- [ ] E1.1: Experiment name [experiment, P1, deps: F1.2]
```

Status derivation:
- `[x]` = done
- `[ ]` + all deps done = ready
- `[ ]` + any dep not done = blocked
- `[decision: ...]` = spike completed
- `[result: ...]` = experiment completed
- Module `[status: X]` in header tracks module-level state

## .current-work.md Format

```markdown
# Current Work: {ID} {Name}

## Type: {feat|infra|spike|migration|experiment}
## Verification: {strategy}

## Description
...

## Acceptance Criteria (for feat/infra)
- criterion 1
- criterion 2

## Questions to Answer (for spike)
- question 1
- question 2

## Hypothesis (for experiment)
...

## Evaluation Metrics (for experiment)
...

## Current Behavior (for migration)
...

## Target Behavior (for migration)
...

## Rollback Plan (for migration)
...

## Related Specs
- [link to spec](...)

## Architecture Constraints
...

## Asset Manifest (if applicable)
| Asset | Type | Status | Notes |
|-------|------|--------|-------|

## Execution Plan
[link to exec-plan](...)
```

The sections present depend on the work item type. Don't include empty sections.
