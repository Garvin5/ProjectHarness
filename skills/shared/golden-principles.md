# Golden Principles

Cross-cutting invariants enforced mechanically across all skills. These are not suggestions — they are guarantees maintained by hooks, skills, and periodic health checks.

## The Principles

### 1. CLAUDE.md is a Derived Artifact

CLAUDE.md files are never hand-written. Their content is derived from the project's source of truth:

- **Root CLAUDE.md**: Navigation table pointing to `docs/project/` files. Generated by `/project-init`, rarely changes.
- **Subdirectory CLAUDE.md**: Derived from `architecture.md` (domain conventions) + `specs/` (current behavior). Content reflects what's true NOW, not what was planned.

**Why:** Hand-written CLAUDE.md drifts from reality. Derived CLAUDE.md stays current because it's regenerated from the source of truth.

**Enforcement:**
- Skills that modify `architecture.md` or `specs/` must check and refresh affected CLAUDE.md files
- PreToolUse hook warns on commit if root CLAUDE.md navigation doesn't match actual `docs/project/` structure
- `/project-health` scans all CLAUDE.md files for staleness

### 2. Spec Coverage

Every completed feature-type work item must have a corresponding spec in `docs/project/specs/`.

**Why:** Specs are the system's memory. Without them, the next agent working on a related feature has no idea what the current system does. This leads to broken assumptions and wasted work.

**Enforcement:**
- `/project-done` generates or updates the spec as part of completion flow
- PreToolUse hook warns on commit if completed features lack specs
- `/project-health` reports spec coverage percentage

### 3. Roadmap Consistency

`exec-plans/active/` status must match `roadmap.md` markers. If the roadmap says a feature is done, the exec-plan must agree, and vice versa.

**Why:** Inconsistent state confuses both humans and agents. If the roadmap says "done" but the exec-plan says "in-progress," no one knows the truth.

**Enforcement:**
- `/project-done` updates both atomically
- `/project-health` cross-checks and flags mismatches

### 4. Architecture Boundaries

Code directory structure must follow `architecture.md` domain model. Code in the wrong domain is a red flag.

**Why:** Architecture drift turns a clean system into a ball of mud. Catching it early is cheap; fixing it late is expensive.

**Enforcement:**
- `/project-health` compares actual directory tree against architecture.md
- Brainstorming skill references architecture constraints from `.current-work.md`

### 5. Decisions are Recorded

Significant technical decisions (framework choices, architecture changes, scope cuts, approach selection from spikes) must be written into `decisions/` or the exec-plan's Decision Log.

**Why:** Without decision records, future agents re-debate settled questions. The cost of recording is minutes; the cost of re-debating is hours.

**Enforcement:**
- Spike completion requires a decision record
- `/project-replan` writes decisions when architecture changes
- This is guidance, not mechanically enforced — it relies on skill discipline

### 6. Assets are Tracked

Non-code deliverables (art, audio, models, configs, documentation) appear in `assets/manifest.md` with status.

**Why:** Untracked assets become invisible dependencies. A feature is "done" but can't ship because the placeholder art was never replaced.

**Enforcement:**
- `/project-next` generates an asset manifest section in `.current-work.md` if the feature needs assets
- `/project-done` updates manifest status
- `/project-health` reports placeholder vs final ratios

## How to Apply

Every skill in this harness includes this line:

> After completing your work, check `shared/golden-principles.md` and execute any relevant invariant maintenance.

In practice this means: after modifying any source-of-truth file (`architecture.md`, `specs/`, `roadmap.md`, `exec-plans/`), scan the principles above and perform any triggered maintenance (refresh CLAUDE.md, update manifest, check consistency).

This is a lightweight check, not a bureaucratic process. Most of the time, 0-1 principles are triggered. The point is to make maintenance automatic rather than something you remember to do.
