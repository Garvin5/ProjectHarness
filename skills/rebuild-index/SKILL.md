# Rebuild INDEX

**Announce:** "Rebuilding `docs/knowledge/INDEX.md`."

**Type:** Rigid — same procedure each run; output is stable so diffs are clean.

**Action style:** Silent postcondition. No proposal, no confirmation. INDEX is fully derived from entry frontmatter — regeneration is reversible (just rerun) and harmless.

## What this skill produces

A regenerated `docs/knowledge/INDEX.md` listing every entry under each type directory, with key fields surfaced for skimming.

Layout:

```markdown
# Knowledge Index — <project name>

> Auto-generated by rebuild-index. Do not edit by hand.
> Last regenerated: <ISO timestamp>
> Total entries: <count>  ·  Superseded (hidden): <count>

## gotchas (N)

- [<title>](gotchas/<file>.md) — `<freshness>` · last-validated <YYYY-MM-DD> · tools `[…]` · domains `[…]`
- …

## decisions (N)
…

## conventions (N)
…

## ground-truth (N)
…

## references (N)
…

## patterns (N)
…
```

Entries with `freshness: superseded` (i.e. those moved to `<type>/_superseded/`) are NOT listed — they are tallied in the header only.

## Procedure

### Step 1 — Locate

Project root is the current git root. KB root is `<project-root>/docs/knowledge/`.

If `docs/knowledge/` does not exist, exit with message "no KB found; nothing to index" and do nothing.

### Step 2 — Run the script

```bash
node "<HARNESS-ROOT>/skills/rebuild-index/scripts/rebuild-index.mjs" "<project-root>"
```

`<HARNESS-ROOT>` is the directory where the harness plugin is installed. When running from inside the harness repo itself, that's the repo root.

The script:

1. Reads `<project-root>/docs/knowledge/manifest.yaml` to extract project name (fallback: directory basename)
2. Globs `<project-root>/docs/knowledge/<type>/*.md`, excluding `_superseded/` subdirs
3. Extracts these fields from each entry's frontmatter via regex:
   - `type` (from frontmatter or directory name)
   - `freshness`
   - `last-validated`
   - `applies-to.tools` and `applies-to.domains`
4. Extracts the first `# <title>` line from the body
5. Groups by type, sorts by filename within each group
6. Writes `<project-root>/docs/knowledge/INDEX.md`
7. Prints `INDEX rebuilt: N entries (M superseded hidden)` to stdout

### Step 3 — Surface

Report a single-line confirmation to the user with the count.

## Constraints

- **Read-only on entry files.** This skill only writes `INDEX.md`.
- **Stable ordering.** Sort by filename within each group so re-runs produce diff-friendly output.
- **No partial updates.** Always full regeneration.
- **Skip `_superseded/`.** Never list superseded entries; tally them in the header for visibility.
- **Tolerate missing frontmatter fields.** If an entry lacks a required field, list it with `<missing>` placeholders rather than crash. `validate-knowledge` is responsible for flagging the malformed entry.

## When NOT to use

- The project has no `docs/knowledge/`. Don't auto-create it.
- The user just wants to read the index. Use `Read docs/knowledge/INDEX.md` directly.
- The user wants to query for entries by tag. That's `consult-knowledge`'s job, not INDEX rebuild.
